## nameHZ

**ИИ-ассистент для врачей и ординаторов, помогающий суммировать, находить и применять знания из учебных медицинских материалов**

## Краткое описание

Проект представляет собой ИИ-ассистента для работы с медицинскими учебными материалами: лекциями, PDF-файлами, расшифровками видеоуроков, презентациями и текстовыми документами.

Система помогает пользователю быстро находить нужную информацию в материалах, получать структурированные объяснения, формировать краткие конспекты, повторять ключевые темы и разбирать учебные клинические случаи с опорой на загруженные источники.

Проект не предназначен для постановки диагноза, назначения лечения или замены врачебного решения. Основной фокус — обучение, повторение материала и поддержка клинического мышления в образовательном формате.

---

## Проблема

Врачи, ординаторы и студенты-медики работают с большим объёмом учебных и клинических материалов: лекциями, презентациями, методичками, клиническими рекомендациями, PDF-документами и видеозаписями занятий.

Основные трудности:

- материалы часто распределены по разным файлам и форматам;
- поиск нужного фрагмента занимает много времени;
- видеолекции сложно быстро пересматривать точечно;
- после занятия трудно быстро восстановить структуру темы;
- сложно выделить главное из большого объёма информации;
- при разборе учебных клинических случаев важно не просто вспомнить факт, а понять логику рассуждения;
- обычный поиск по ключевым словам плохо работает, если пользователь формулирует вопрос иначе, чем это написано в материале.

---

## Цель проекта

Создать ИИ-ассистента, который помогает врачам и ординаторам эффективно работать с учебными медицинскими материалами: искать информацию, получать краткие и структурированные объяснения, формировать конспекты, генерировать вопросы для самопроверки и разбирать учебные клинические случаи с обязательной ссылкой на исходные материалы.

---

## Целевая аудитория

Основные пользователи:

- врачи, проходящие дополнительное обучение;
- ординаторы;
- студенты медицинских вузов старших курсов;
- преподаватели медицинских дисциплин;
- участники медицинских курсов и образовательных программ.

---

## Основной функционал

### 1. Загрузка учебных материалов

Пользователь может загружать:

- PDF-файлы;
- текстовые документы;
- презентации;
- расшифровки лекций;
- видеозаписи занятий.

Для видео система выполняет speech-to-text, то есть перевод речи в текст.

---

### 2. Поиск по материалам

Пользователь задаёт вопрос естественным языком, например:

> Какие основные причины боли в груди разбирались на лекции?

Система ищет релевантные фрагменты в загруженных материалах и возвращает ответ с указанием источников.

---

### 3. Суммаризация материалов

Система может формировать:

- краткий конспект занятия;
- подробный конспект;
- список ключевых терминов;
- основные тезисы;
- структуру темы;
- список важных клинических признаков;
- краткое повторение перед зачётом или экзаменом.

---

### 4. Объяснение сложных тем

Пользователь может попросить объяснить отдельный фрагмент, термин или тему.

Система должна уметь отвечать в разных форматах:
- кратко;
- подробно;
- простым языком;
- профессиональным медицинским языком;
- в виде таблицы;
- в виде алгоритма;
- в формате “признак → значение → почему важно”.

---

### 5. Разбор учебных клинических случаев

Пользователь может ввести учебный клинический случай.

Система помогает разобрать его в образовательном формате:

- выделяет ключевые симптомы и признаки;
- показывает, какие данные важны;
- предлагает возможные направления рассуждения;
- формирует дифференциальный ряд;
- объясняет аргументы “за” и “против”;
- предлагает уточняющие вопросы;
- указывает, какие фрагменты учебных материалов связаны с этим случаем.

Важно: система не ставит диагноз, а помогает разобрать учебный кейс.

---

### 6. Ответы с источниками

Каждый содержательный ответ должен сопровождаться ссылками на исходные материалы:

- название документа;
- страница;
- раздел;
- временная метка видео, если доступна;
- фрагмент текста, на который опирается ответ.

Если в базе знаний нет достаточной информации, система должна явно сообщить об этом.

---

## Предполагаемый технический подход

Основой проекта является RAG-подход.

RAG — retrieval-augmented generation, то есть генерация ответа с предварительным поиском по базе знаний.

Общий пайплайн:

1. Пользователь загружает учебные материалы.
2. Система извлекает из них текст.
3. Текст очищается и разбивается на смысловые фрагменты.
4. Для каждого фрагмента строится embedding.
5. Фрагменты сохраняются в векторную базу данных.
6. При вопросе пользователя система ищет релевантные фрагменты.
7. Найденные фрагменты передаются в LLM.
8. LLM формирует ответ строго на основе найденных источников.

Embedding — векторное представление текста. Оно позволяет искать не только по точному совпадению слов, но и по смысловой близости.

## ML-компоненты проекта

### 1. Speech-to-text

Используется для преобразования видеолекций в текст.

Возможный инструмент: Whisper или аналогичная модель.

---

### 2. Text preprocessing

Предобработка текста:

- очистка мусора;
- удаление повторов;
- исправление очевидных ошибок распознавания;
- разбиение по темам;
- сохранение структуры документа;
- привязка текста к источнику.

---

### 3. Chunking

Chunking — разбиение текста на фрагменты.

Цель: сделать такие фрагменты, которые достаточно маленькие для поиска, но не теряют смысл.

Пример фрагмента:

- определение заболевания;
- симптомы;
- диагностические критерии;
- алгоритм;
- клинический пример;
- блок “ошибки и исключения”.

---

### 4. Embedding model

Модель переводит каждый текстовый фрагмент в вектор.

Затем по запросу пользователя строится embedding запроса, и система ищет похожие фрагменты в базе.

---

### 5. Vector database

Векторная база хранит embeddings и метаданные:

- источник;
- название файла;
- страница;
- тема;
- номер чанка;
- временная метка видео;
- исходный текст.

Возможные инструменты:

- Qdrant;
- FAISS;
- Chroma;
- pgvector.

---

### 6. Retrieval

Retrieval — поиск релевантных фрагментов по запросу пользователя.

На этом этапе система выбирает top-k фрагментов, которые наиболее похожи на вопрос.

Top-k означает “первые k результатов”, например top-5 или top-10.

---

### 7. Reranking

Reranking — повторная сортировка найденных фрагментов более точной моделью.

Это нужно, чтобы убрать нерелевантные куски и оставить только те, которые действительно помогают ответить на вопрос.

---

### 8. Structured output generation

Ответ генерируется не свободно, а по строгому шаблону.

Например:

- краткий ответ;
- подробное объяснение;
- ключевые тезисы;
- что важно запомнить;
- вопросы для самопроверки;
- источники;
- ограничения ответа.

---

### 9. Structured disease-symptom knowledge base

На следующих этапах можно добавить структурированную базу знаний:

- заболевания;
- симптомы;
- типичные признаки;
- атипичные признаки;
- возрастные особенности;
- красные флаги;
- уточняющие вопросы;
- дифференциальные диагнозы.

Эта база не заменяет RAG, а дополняет его. Она нужна для учебного разбора клинических кейсов и более строгого формирования дифференциального ряда.

---

### 10. Scoring

Scoring — начисление баллов гипотезам.

В учебных клинических кейсах система может оценивать, какие заболевания лучше соответствуют описанию случая, на основе симптомов, возраста, признаков и красных флагов.

Результат scoring используется не как окончательный диагноз, а как учебная подсказка для разбора случая.

---

## Формат ответа системы

Для обычного вопроса:

1. Краткий ответ.
2. Подробное объяснение.
3. Ключевые термины.
4. Что важно запомнить.
5. Источники.

Для суммаризации лекции:

1. Тема занятия.
2. Основные блоки.
3. Ключевые тезисы.
4. Важные определения.
5. Клинические примеры.
6. Вопросы для самопроверки.

Для учебного клинического случая:

1. Выделенные симптомы и признаки.
2. Недостающие данные.
3. Возможные направления рассуждения.
4. Аргументы “за” и “против”.
5. Уточняющие вопросы.
6. Связанные фрагменты учебных материалов.
7. Ограничение: система не ставит диагноз.

## MVP

### MVP-1: RAG по текстовым материалам

Функции:

- загрузка PDF и текстовых файлов;
- разбиение документов на фрагменты;
- построение embeddings;
- поиск по материалам;
- генерация ответа с источниками.

Цель MVP-1: проверить, может ли система качественно отвечать на вопросы по загруженным учебным материалам.

---

### MVP-2: Работа с видео

Функции:
- загрузка видео;
- автоматическая расшифровка;
- привязка текста к временным меткам;
- поиск по видеолекциям;
- ответы со ссылками на таймкоды.

---

### MVP-3: Суммаризация и самопроверка

Функции:
- краткий конспект лекции;
- список ключевых терминов;
- генерация тестов;
- генерация карточек;
- вопросы для повторения.

---

### MVP-4: Учебные клинические кейсы

Функции:

- выделение симптомов и признаков из кейса;
- генерация уточняющих вопросов;

- аргументы “за” и “против”;
- связь кейса с материалами занятий.

## Метрики качества

### Для поиска по материалам

- Recall@k — попал ли нужный фрагмент в первые k результатов.
- Precision@k — сколько из первых k найденных фрагментов действительно релевантны.
- MRR — насколько высоко в выдаче находится первый правильный фрагмент.

### Для генерации ответа

- доля ответов с корректными источниками;
- доля ответов без неподтверждённых утверждений;
- полнота ответа;
- понятность ответа;
- соответствие заданному формату.

### Для суммаризации

- покрытие ключевых тем;
- отсутствие важных пропусков;
- отсутствие искажений;
- удобство конспекта для повторения.

### Для учебных клинических кейсов

- корректность выделения симптомов;
- качество уточняющих вопросов;
- наличие правильного диагноза в top-k учебных гипотез;
- корректность аргументов “за” и “против”;
- отсутствие опасных медицинских утверждений.

## Ограничения проекта

Система:

- не ставит окончательный диагноз;
- не назначает лечение;
- не заменяет врача;
- не используется как самостоятельная медицинская система принятия решений;
- работает только с теми знаниями, которые есть в загруженных материалах;
- может ошибаться при плохом качестве исходных данных;
- требует экспертной проверки медицинского содержания.

---

## Потенциальные риски

- ошибки распознавания речи в видеолекциях;
- плохое качество PDF или презентаций;
- неполные или устаревшие учебные материалы;
- некорректный поиск релевантных фрагментов;
- генерация утверждений без достаточной опоры на источники;
- смешение учебного разбора и реальной медицинской рекомендации;
- неправильная интерпретация клинического случая пользователем.

---

## Способы снижения рисков

- обязательные ссылки на источники;
- запрет на ответы без найденного контекста;
- строгий формат ответа;
- явное указание ограничений;
- ручная проверка медицинских материалов;
- отдельная маркировка учебных клинических кейсов;
- тестирование на заранее подготовленных вопросах;
- экспертная оценка ответов врачом или преподавателем.

---

## Ожидаемый результат

В результате проекта должен получиться ИИ-ассистент, который позволяет пользователю загружать медицинские учебные материалы и работать с ними через естественный язык.

Система должна помогать:

- быстро находить информацию;
- получать структурированные объяснения;
- формировать конспекты;
- повторять материал;
- генерировать вопросы для самопроверки;
- разбирать учебные клинические случаи;
- видеть, на какие источники опирается каждый ответ.

Главная ценность проекта — не автоматическая диагностика, а ускорение и структурирование работы с медицинскими знаниями.
