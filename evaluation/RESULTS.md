# Результаты retrieval-эксперимента

Эксперимент проведён 20 июля 2026 года. Его задача — выбрать retrieval-конфигурацию для рабочего MVP на основании измерений, а не предположения о том, что более сложный pipeline обязательно будет лучше.

## Условия

- 3 медицинских материала MedQuAD/NIDDK;
- 18 размеченных фрагментов;
- 60 вопросов: 15 исходных и 45 детерминированных переформулировок;
- dense-модель: `intfloat/multilingual-e5-large`;
- reranker: `BAAI/bge-reranker-v2-m3`;
- выполнение на CPU внутри Docker;
- latency измерялась после прогрева модели и не включает загрузку весов и построение индекса.

Полные машинные результаты находятся в:

- [`results/benchmark_results.csv`](results/benchmark_results.csv);
- [`results/per_question.jsonl`](results/per_question.jsonl);
- [`results/run_metadata.json`](results/run_metadata.json).

## Итоговая таблица

| Конфигурация | Recall@5 | MRR | Top-1 gold accuracy | Latency p50 |
|---|---:|---:|---:|---:|
| BM25 | 0.733 | 0.410 | 18.3% | 0.0001 сек. |
| **Dense** | **0.983** | **0.865** | **78.3%** | **0.297 сек.** |
| Dense + reranker | 1.000 | 0.715 | 58.3% | 5.525 сек. |

`Top-1 gold accuracy` означает долю вопросов, для которых первый найденный фрагмент входит в размеченный gold-набор. Это строгая retrieval-метрика, а не экспертная оценка полного сгенерированного ответа.

В абсолютных значениях:

- BM25 нашёл gold-фрагмент в top-5 для 44 из 60 вопросов и поставил его первым для 11 из 60;
- Dense нашёл gold-фрагмент в top-5 для 59 из 60 вопросов и поставил его первым для 47 из 60;
- Dense + reranker нашёл gold-фрагмент в top-5 для 60 из 60 вопросов, но поставил его первым только для 35 из 60.

## Принятое решение

**Основная конфигурация проекта — Dense retrieval без reranker.**

Причины:

1. Dense показал максимальный MRR: `0.865`.
2. Dense показал максимальную точность первого фрагмента: `78.3%`.
3. Recall@5 уже почти предельный: `0.983`, то есть пропущен только один вопрос из 60.
4. Медианная задержка составляет около `0.297` секунды и подходит для интерактивного локального приложения.
5. Reranker увеличил Recall@5 только на один вопрос, но ухудшил MRR и top-1 accuracy.
6. Reranker увеличил медианную задержку примерно в 18.6 раза: с `0.297` до `5.525` секунды.

Следовательно, усложнение pipeline в текущих условиях не оправдано.

## Как интерпретировать результат reranker

Эксперимент не доказывает, что `bge-reranker-v2-m3` является плохой моделью. Он показывает более узкий факт:

> на текущем небольшом корпусе, текущих gold-фрагментах, текущем способе chunking и CPU-инференсе этот reranker ухудшает наиболее важные для приложения метрики и резко увеличивает задержку.

Высокий `Recall@5 = 1.0` одновременно с более низкими MRR и top-1 accuracy означает, что правильный фрагмент обычно остаётся среди первых результатов, но reranker часто опускает его с первого места ниже.

Возможные причины:

- корпус содержит только 18 фрагментов, поэтому dense retrieval уже почти решает задачу;
- 45 вопросов являются шаблонными переформулировками;
- часть длинных фрагментов может обрезаться на входе cross-encoder;
- строгая gold-разметка может не учитывать альтернативные фрагменты, которые тоже содержат полезный ответ;
- `candidate_k=20` фактически передаёт reranker почти весь корпус и создаёт большую CPU-задержку.

## Статус reranker в приложении

Reranker сохранён как экспериментальная опция, но отключён по умолчанию.

Глобальное включение:

```dotenv
RERANKER_ENABLED=true
```

Для конкретного запроса также нужно передать:

```json
{
  "use_reranker": true
}
```

При `RERANKER_ENABLED=false` запрос обрабатывается обычным dense retrieval, даже если клиент передал `use_reranker=true`; поле `rerank_score` остаётся `null`.

Это позволяет повторять эксперименты, не делая более медленную и менее точную конфигурацию стандартной.

## Ограничения эксперимента

- набор небольшой;
- материалы и вопросы англоязычные;
- 45 из 60 вопросов созданы по фиксированным шаблонам;
- один MedQuAD `Answer` используется как один gold-фрагмент;
- некоторые gold-фрагменты длиннее production-чанков приложения;
- измерения latency относятся к конкретному CPU/Docker-окружению;
- эксперимент оценивает retrieval, а не медицинскую корректность сгенерированного ответа.

## Следующие шаги

Перед повторным рассмотрением reranker следует:

1. собрать более крупный набор на реальных загруженных лекциях и PDF;
2. добавить вручную проверенные русскоязычные вопросы;
3. проверить `candidate_k` равный 5 и 10;
4. проанализировать вопросы, где Dense имеет rank 1, а reranker опускает gold-фрагмент;
5. оценить альтернативные reranker-модели и более короткие production-чанки;
6. сравнивать конфигурации на одном и том же устройстве и наборе данных.
